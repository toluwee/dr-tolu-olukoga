The Interview Game Changed: A Senior Engineer's Guide to Proving You Can Think, Not Just Type

March 1, 2026 • 7 min read
Career Development AI Tools Interviews Software Engineering

Last month, I interviewed at three different companies. Each one gave me the same task: review code that ChatGPT had generated and tell them what was wrong with it.

Not "write me a function." Not "explain this algorithm." Not even "design this system from scratch."

Just: "Here's code that passes our unit tests. Would you ship it to production? Why or why not?"

This is the new reality. I recently read an article from Built In describing this shift from the employer's side, and it crystallized something I've been experiencing: we're not being hired to write code anymore. We're being hired to decide whether code should exist.

After building agentic AI systems and watching junior engineers learn to use Claude and ChatGPT before they learn to use Git, I've realized the skill gap isn't syntax. It's judgment. Here's what I've learned about preparing for this new interview landscape.

The Wake-Up Call: My First "Audit Interview"

Six months ago, I walked into what I thought would be a standard system design interview. The interviewer pulled up a Python service that handled user authentication. Clean code. Good variable names. Type hints everywhere. It even had docstrings.

"ChatGPT wrote this," they said. "You have 20 minutes. Tell me what happens when we deploy it."

I panicked. This wasn't the game I'd practiced. I'd spent weeks on LeetCode. I could reverse a binary tree in my sleep. But nobody had asked me to be a code reviewer for an AI.

I fumbled through it, caught the obvious SQL injection vulnerability, mentioned something about rate limiting, and called it done. I didn't get the job.

The feedback? "You found the security issue, but you missed that this would fall over at scale. You didn't ask about our user base size. You didn't question the session storage strategy. You treated it like a code review, not a production readiness assessment."

That was my wake-up call.

What I Changed: The Three-Layer Review Framework

I stopped preparing like I was going to write code. I started preparing like I was going to be the last line of defense before production.

Here's the mental model I now use for every piece of code I see in an interview:

Layer 1: The "Will This Break?" Scan

Before I say anything clever, I ask: At what scale does this stop working?

This isn't about complexity analysis. This is about production reality. I learned to spot the patterns:

  • Unbounded loops over database results (the classic N+1 killer)
  • In-memory caches without eviction policies (the memory leak time bomb)
  • Synchronous external API calls in the request path (the latency disaster)
  • Missing retry logic with exponential backoff (the cascade failure starter kit)

In my last interview, they showed me a file processing service. The code was beautiful. It also loaded entire CSV files into memory before processing them. I asked: "What's the largest file we expect?" They said 10MB. I said: "What happens when someone uploads a 2GB file?"

They smiled. That was the right question.

Layer 2: The "What Am I Not Seeing?" Investigation

AI-generated code has a tell: it solves the problem you asked for, and nothing else. It doesn't know your infrastructure. It doesn't know your constraints. It doesn't know that your legacy database can't handle more than 100 connections.

I started asking context questions before I even looked at the code:

  • "What's the expected request rate?"
  • "How is this service deployed? Containers? Serverless?"
  • "What's the downstream dependency? Can it handle our peak load?"
  • "Do we have SLAs I should know about?"

Last week, I reviewed code that queried an external API. Clean code. Proper error handling. I asked: "Do we pay per API call?" They said yes. The code was making 50 calls per user request. That would have cost them $30K/month.

One question saved them a budget crisis.

Layer 3: The "How Would This Actually Ship?" Reality Check

This is where I separate myself from the people who just learned to code last year. Production systems aren't about perfect code. They're about trade-offs, technical debt, and shipping safely.

When I review code now, I explicitly talk about:

  • What I'd fix immediately: The stuff that will cause an outage
  • What I'd log as a ticket: The stuff that's inefficient but not dangerous
  • What I'd leave alone: The stuff that's ugly but works

In one interview, they gave me code with three problems: terrible variable names, a potential memory leak, and a race condition in the payment processing logic.

I said: "The race condition is a P0. Users could get double-charged. I'd fix that before we ship. The memory leak is a P1. We can monitor it and address it in the next sprint. The variable names are tech debt. I'd leave them and let the next refactor clean them up."

They hired me.

The AI Partner Test: Show You're Driving, Not Just Riding

Here's what surprised me: interviewers don't care if you use AI during the interview. They care whether you trust it blindly or verify everything it says.

In my last interview, I was allowed to use ChatGPT. I did. But I narrated my process:

"I'm going to ask ChatGPT to suggest a caching strategy. [Types prompt] Okay, it's recommending Redis with a 5-minute TTL. That makes sense for session data, but I want to double-check the eviction policy. [Reads docs] Actually, I'm going to modify this to use LRU eviction instead of the default TTL because our access patterns are uneven."

The interviewer stopped me. "This is exactly what we want to see. You're using the tool, but you're still thinking."

That's the skill. Treating AI like a junior engineer who gives you a starting point, not an oracle who gives you the answer.

What Junior Engineers Get Wrong (And How to Fix It)

If you're early in your career, here's the trap: you think the interview is about showing you know things. It's not. It's about showing you know what you don't know.

The best junior engineer I ever interviewed said this: "I don't recognize this caching library. Can you tell me if it's thread-safe? Because if not, we'll have a problem when we scale horizontally."

They didn't know the answer. But they knew the question mattered. That's the skill.

When you're preparing:

  • Stop memorizing algorithms. Start reading post-mortems.
  • Stop solving toy problems. Start reviewing production code on GitHub.
  • Stop trying to write perfect code. Start asking what "perfect" means in context.

The interview isn't testing if you can code. It's testing if you can think like someone who has to maintain the code you wrote at 2 AM six months from now.

My Current Interview Prep Routine

I don't practice coding anymore. I practice reviewing.

Once a week, I:

  1. Pick a popular open-source repo
  2. Find a recent merged PR
  3. Review it like I'm the gatekeeper: What would break? What's the cost? What's the trade-off?
  4. Read the actual PR comments to see what the maintainers caught that I missed

This has made me better at interviews and better at my actual job.

The Bottom Line

The game changed. Interviewers aren't hiring code generators anymore. They're hiring people who can look at auto-generated code and say: "This will cost us $50K/month in Lambda invocations" or "This will fall over when we hit 10K users" or "This is fine for now, but we should refactor it in Q3."

If you can do that, you're valuable. If you can't, you're competing with ChatGPT.

And ChatGPT works for free.
