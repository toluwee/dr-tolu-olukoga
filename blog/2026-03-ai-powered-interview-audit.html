The Interview Game Changed: What Flora's Journey Taught Me About Proving You Can Think, Not Just Type

March 1, 2026 • 8 min read
Career Development AI Tools Interviews Software Engineering


Engineer examining AI-generated code with a magnifying glass in the interview audit factory. Illustration by nano banana 2.

Last month, my colleague Flora interviewed at three different companies. Each one gave her the same task: review code that ChatGPT had generated and tell them what was wrong with it.

Not "write me a function." Not "explain this algorithm." Not even "design this system from scratch."

Just: "Here's code that passes our unit tests. Would you ship it to production? Why or why not?"

When Flora told me about this, it clicked. I recently read an article from Built In describing this shift from the employer's side, and watching Flora's interview journey crystallized something we've both been seeing in production: companies aren't hiring code generators anymore. They're hiring people who can decide whether code should exist.

After years building agentic AI systems and watching junior engineers learn to use Claude and ChatGPT before they learn to use Git, the skill gap is clear. It isn't syntax. It's judgment. Here's what Flora's experience taught both of us about the new interview landscape.

The Wake-Up Call: Flora's First "Audit Interview"

Six months ago, Flora walked into what she thought would be a standard system design interview. The interviewer pulled up a Python service that handled user authentication. Clean code. Good variable names. Type hints everywhere. It even had docstrings.

"ChatGPT wrote this," they said. "You have 20 minutes. Tell me what happens when we deploy it."

Flora panicked. This wasn't the game she'd practiced. She'd spent weeks on LeetCode. She could reverse a binary tree in her sleep. But nobody had asked her to be a code reviewer for an AI.

She fumbled through it, caught the obvious SQL injection vulnerability, mentioned something about rate limiting, and called it done. She didn't get the job.

The feedback? "You found the security issue, but you missed that this would fall over at scale. You didn't ask about our user base size. You didn't question the session storage strategy. You treated it like a code review, not a production readiness assessment."

That was Flora's wake-up call.

What Flora Changed: The Three-Layer Review Framework

Flora stopped preparing like she was going to write code. She started preparing like she was going to be the last line of defense before production.

Here's the mental model she developed for every piece of code she saw in interviews:


Engineer wielding constraints and best practices against a malfunctioning AI code bot. Illustration by nano banana 2.

Layer 1: The "Will This Break?" Scan

Before saying anything clever, Flora learned to ask: At what scale does this stop working?

This isn't about complexity analysis. This is about production reality. She trained herself to spot the patterns:

  • Unbounded loops over database results (the classic N+1 killer)
  • In-memory caches without eviction policies (the memory leak time bomb)
  • Synchronous external API calls in the request path (the latency disaster)
  • Missing retry logic with exponential backoff (the cascade failure starter kit)

In her second round of interviews, they showed her a file processing service. The code was beautiful. It also loaded entire CSV files into memory before processing them. Flora asked: "What's the largest file we expect?" They said 10MB. She followed up: "What happens when someone uploads a 2GB file?"

They smiled. That was the right question.

Layer 2: The "What Am I Not Seeing?" Investigation

Flora noticed that AI-generated code has a tell: it solves the problem you asked for, and nothing else. It doesn't know your infrastructure. It doesn't know your constraints. It doesn't know that your legacy database can't handle more than 100 connections.

She started asking context questions before she even looked at the code:

  • "What's the expected request rate?"
  • "How is this service deployed? Containers? Serverless?"
  • "What's the downstream dependency? Can it handle our peak load?"
  • "Do we have SLAs I should know about?"


Engineer analyzing system architecture diagrams and weighing technical trade-offs with cost considerations. Illustration by nano banana 2.

In one interview, Flora reviewed code that queried an external API. Clean code. Proper error handling. She asked: "Do we pay per API call?" They said yes. The code was making 50 calls per user request. That would have cost them $30K/month.

One question saved them a budget crisis.

Layer 3: The "How Would This Actually Ship?" Reality Check

This is where Flora separated herself from candidates who'd just learned to code. Production systems aren't about perfect code. They're about trade-offs, technical debt, and shipping safely.

When reviewing code, she explicitly discussed:

  • What to fix immediately: The stuff that will cause an outage
  • What to log as a ticket: The stuff that's inefficient but not dangerous
  • What to leave alone: The stuff that's ugly but works

In her final interview at the company where she's now working, they gave her code with three problems: terrible variable names, a potential memory leak, and a race condition in the payment processing logic.

Flora said: "The race condition is a P0. Users could get double-charged. I'd fix that before we ship. The memory leak is a P1. We can monitor it and address it in the next sprint. The variable names are tech debt. I'd leave them and let the next refactor clean them up."

They hired her on the spot.

The AI Partner Test: Show You're Driving, Not Just Riding

Here's what surprised Flora: interviewers don't care if you use AI during the interview. They care whether you trust it blindly or verify everything it says.


Engineer patching critical system issues including N+1 queries, memory leaks, and batch fetch problems. Illustration by nano banana 2.

In one of her later interviews, Flora was explicitly allowed to use ChatGPT. She did. But she narrated her process out loud:

"I'm going to ask ChatGPT to suggest a caching strategy. [Types prompt] Okay, it's recommending Redis with a 5-minute TTL. That makes sense for session data, but I want to double-check the eviction policy. [Reads docs] Actually, I'm going to modify this to use LRU eviction instead of the default TTL because our access patterns are uneven."

The interviewer stopped her. "This is exactly what we want to see. You're using the tool, but you're still thinking."

That's the skill. Treating AI like a junior engineer who gives you a starting point, not an oracle who gives you the answer.

What Junior Engineers Get Wrong (And How Flora Fixed It)

Flora told me the trap for early-career engineers: they think the interview is about showing they know things. It's not. It's about showing they know what they don't know.

She shared an example from a junior engineer she interviewed recently who said: "I don't recognize this caching library. Can you tell me if it's thread-safe? Because if not, we'll have a problem when we scale horizontally."

They didn't know the answer. But they knew the question mattered. That's the skill.

Flora's advice for junior engineers preparing:

  • Stop memorizing algorithms. Start reading post-mortems.
  • Stop solving toy problems. Start reviewing production code on GitHub.
  • Stop trying to write perfect code. Start asking what "perfect" means in context.

The interview isn't testing if you can code. It's testing if you can think like someone who has to maintain the code you wrote at 2 AM six months from now.

Flora's Interview Prep Routine

After her initial failures, Flora completely changed her approach. She doesn't practice coding anymore. She practices reviewing.

Her weekly routine:

  1. Pick a popular open-source repo
  2. Find a recent merged PR
  3. Review it like she's the gatekeeper: What would break? What's the cost? What's the trade-off?
  4. Read the actual PR comments to see what the maintainers caught that she missed

She told me this preparation made her better at interviews and better at her actual job.

The Bottom Line

Watching Flora's journey made one thing clear: the game has changed. Interviewers aren't hiring code generators anymore. They're hiring people who can look at auto-generated code and say: "This will cost us $50K/month in Lambda invocations" or "This will fall over when we hit 10K users" or "This is fine for now, but we should refactor it in Q3."

If you can do that, you're valuable. If you can't, you're competing with ChatGPT.

And ChatGPT works for free.

Flora's now thriving in her new role, and I've adopted many of her preparation strategies for my own work. The skill that got her hired wasn't coding—it was judgment. That's what companies are paying for now.
