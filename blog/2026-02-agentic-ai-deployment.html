Building a Production Agentic AI System: Lessons from Automating Kubernetes Deployments

February 20, 2026 • 12 min read
Agentic AI LangGraph Kubernetes Production ML
TL;DR: I built a 22-node LangGraph workflow that accepts natural language ("Deploy patient-dashboard to dev") and orchestrates the full Kubernetes deployment lifecycle across GitLab, Terraform, ArgoCD, and AWS — reducing deployment complexity by 72% and cutting time from 2-4 hours to 30 minutes. Here's what building a production-grade agentic system actually taught me.

The Problem: Five Systems, One Deployment

Deploying a Streamlit application to a Kubernetes cluster inside a regulated healthcare environment meant touching five distinct systems in the right order:

  1. GitLab — write a Dockerfile, CI/CD pipeline, and Helm chart; commit to the right branch
  2. Terraform — provision an SSL certificate via AWS ACM; wait for DNS validation
  3. AWS Route53 — configure DNS records pointing at the application load balancer
  4. ArgoCD — sync the Kubernetes application from the Git source of truth
  5. AWS — verify the ALB hostname, confirm certificate issuance, validate health

Each step had its own CLI, its own credential chain, and its own failure modes. Getting all of this right once required deep familiarity with the infrastructure. Getting it right repeatedly — for every new app, every new environment — required a specialist.

The goal was clear: a data scientist should be able to type "Deploy my-dashboard to dev" and walk away. No Kubernetes knowledge required.

The 22-node LangGraph StateGraph is the core. Each node is a pure function that takes a DeploymentState TypedDict and returns an updated state. LangGraph handles the graph traversal, conditional edge routing, and — crucially — checkpoint persistence.

Why LangGraph (and Not Just Chained Prompts)

Before settling on LangGraph, the natural instinct is to reach for something simpler: a Python function that calls the OpenAI API, parses the response, and calls the next function. That works for demos. It doesn't work in production.

Here's why the state machine model matters for this kind of system:

Workflows have approval gates. Terraform pipelines in a regulated environment require manual approval before running. The system can't simply block a thread waiting for a human to click a button — that could be minutes or hours. A state machine can serialize its state to disk, terminate cleanly, and resume exactly where it left off when conditions are met.

Errors require routing, not just retry. A "folder not found" in GitLab requires a different response than a transient API timeout. LangGraph's conditional edge routing lets you define explicit recovery strategies per error class, rather than wrapping everything in try/except and hoping for the best.

Progress must be observable. When a user submits a deployment request, they need to see what's happening. LangGraph's astream() yields state updates node-by-node, which the Streamlit UI maps to a real-time progress bar. This is impossible with a monolithic function.

State must survive restarts. Kubernetes pods get evicted. Network connections drop. The SQLite checkpoint database means a deployment that starts on a Monday can survive a pod restart and resume Tuesday without any lost work.

Five Production Challenges (and How We Solved Them)

1. The Waiting Problem

Terraform pipelines in this environment require a human to click a "play" button in GitLab before terraform jobs execute. This is a deliberate security control — we can't auto-trigger it.

The naive solution is to poll indefinitely in a loop. The problem is that "indefinitely" might be 45 minutes, and you can't hold a thread open that long in a production system.

Solution: Workflow pause/resume via LangGraph checkpoints.

When the workflow hits a Terraform approval gate, it serializes its full state to SQLite and terminates. A background polling service reads the checkpoint database every 60 seconds, queries the GitLab API for pipeline status, and calls orchestrator.resume(thread_id) when the pipeline completes. The workflow resumes from the exact node it paused at, with all prior state intact.

def wait_for_certificate_pipeline_node(state: DeploymentState) -> DeploymentState:
    pipeline_url = state.get("cert_pipeline_url")
    status = check_gitlab_pipeline_status(pipeline_url)

    if status == "manual":
        return {**state, "paused": True, "pause_reason": "Waiting for manual pipeline trigger"}
    elif status == "success":
        return {**state, "paused": False, "cert_pipeline_complete": True}
    else:
        return {**state, "paused": True}  # Still running, will re-check

The polling service picks this up, sees paused: True, and re-invokes the workflow when the pipeline succeeds.

2. HIPAA-Compliant State Encryption

The checkpoint database contains sensitive data: GitLab tokens, certificate ARNs, deployment configs, and infrastructure details. For a HIPAA-regulated healthcare environment, this data must be encrypted at rest.

Solution: SQLCipher with environment-gated encryption.

In development (RUNTIME_MODE=dev), the database is unencrypted for developer convenience. In production (RUNTIME_MODE=prod), the SQLiteCheckpointer uses SQLCipher with a 256-bit key from AWS Secrets Manager:

if self.runtime_mode == "prod":
    conn.execute(f"PRAGMA key = '{self.encryption_key}'")
    conn.execute("PRAGMA kdf_iter = 64000")
    conn.execute("PRAGMA cipher_page_size = 4096")

The encryption key is generated with openssl rand -hex 32 and stored in AWS Secrets Manager — never in code or environment files.

3. LLM Output Reliability

The generate_configs node asks GPT-4 to write a production-ready Dockerfile, GitLab CI pipeline, and requirements.txt for an application it has never seen. LLMs are creative. You cannot afford creative Dockerfiles in production.

Solution: Constrained templates with LLM-powered substitution, plus a deterministic fallback.

The system provides GPT-4 with rigid template structures and asks it only to fill in the variables it can infer (entry point filename, Python version, framework-specific environment variables). Temperature is set to 0.1. The response is parsed by extracting fenced code blocks.

If the LLM is unavailable or produces malformed output, a stub agent generates safe, deterministic defaults from the repository analysis results. No deployment fails because GPT-4 hallucinated a CMD instruction.

4. Multi-System Error Recovery

With 22 nodes and 5 external systems, the failure space is large. Not all failures are equal:

  • A GitLab "repository not found" means the user needs to fix a typo — there's no point retrying.
  • A GitLab API timeout is transient — retry up to 3 times with backoff.
  • An ArgoCD sync failure may require rollback.

Solution: Standardized error codes with routing-based recovery.

The system defines 15+ error codes (VALIDATION_ERROR, AUTH_ERROR, CONNECTION_ERROR, DEPLOYMENT_FAILED, etc.). The error_recovery_agent_node reads the error code from state and routes to the appropriate path via LangGraph conditional edges.

5. Production Safety Guards

When you build a system that deploys to Kubernetes via natural language, you have to think hard about what happens when a user types "Deploy my-app to production."

Solution: Multiple explicit safety controls.

  • Environment blocklist: Any request containing "production", "prod", "prd", "live", "main", or "master" is rejected at the parse_request node with a clear error message.
  • Repository allowlist: Only GitLab URLs matching GITLAB_ALLOWED_HOSTS are accepted. Prevents arbitrary repo injection.
  • Input sanitization: LLM-generated content and error messages are sanitized before logging.
  • Audit trail: Every significant action is logged with a correlation ID for incident investigation.

Key Lessons Learned

1. Treat your AI workflow like a distributed system, not a script.
Checkpointing, idempotency, and recovery paths are as important in LLM workflows as they are in microservices. Design for failure from the start.

2. Polling is often the right answer.
Webhooks require an exposed endpoint. Polling requires a loop. In an enterprise environment with strict network controls, polling across your own SQLite database is vastly simpler to operate and debug.

3. Temperature 0.1 for infrastructure generation.
Creativity is a liability when the LLM is writing Dockerfiles. Use deterministic settings and constrain the output format strictly. Always have a fallback.

4. State machines make agentic systems auditable.
Every state transition in LangGraph is a discrete, logged event. When something goes wrong (and it will), you can replay the state history and identify exactly where it diverged. This is invaluable for debugging in production.

5. The hardest part isn't the AI.
The genuinely hard problems in this system — checkpoint-based resumption, encrypted state storage, multi-system error recovery — have nothing to do with LLMs. They're distributed systems problems. The LLM is the easy part.

What's Next
